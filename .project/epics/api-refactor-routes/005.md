# Task: Extract chat & push routers

## Description

Refactor service-oriented and event-driven endpoints into dedicated routers. The chat router handles AI model inference and chat messaging, while the push router manages browser push notifications and subscription lifecycle. Both have clear external service boundaries and can be tested with straightforward mocking.

## Technical Details

### Chat Router Implementation

```typescript
// Router exports this function
export async function handleChatRoute(
  req: Request,
  url: URL
): Promise<Response | undefined> {
  // Return undefined for unmapped paths
  // Handle: POST /api/chat, GET /api/models
  // Forward to models API, handle tool calling
}
```

**Endpoints:**
- `POST /api/chat` — Send chat message:
  - Parse request with user identity, message content, model selection
  - Forward to models/AI inference API
  - Handle tool calling requests (may spawn runners, execute code, etc.)
  - Return model response with proper streaming/chunking if needed
  - Error handling for API timeouts, rate limits, invalid models
- `GET /api/models` — List available models:
  - Return array of model metadata (name, id, capabilities, rate limits, etc.)
  - Filter based on user permissions (some models may require subscription)
  - Cache model list (with reasonable TTL)

**Dependencies:**
- Models API client (from `packages/server/src/` or external)
- Auth context from handler.ts
- Tool registry for callable tools

### Push Router Implementation

```typescript
// Router exports this function
export async function handlePushRoute(
  req: Request,
  url: URL
): Promise<Response | undefined> {
  // Return undefined for unmapped paths
  // Handle: GET/POST/PUT /api/push/*
  // Manage subscription lifecycle and VAPID keys
}
```

**Endpoints:**
- `GET /api/push/vapid-public-key` — Retrieve VAPID public key:
  - Return base64-encoded VAPID public key (for client-side service worker setup)
  - No auth required (public key is meant to be public)
- `POST /api/push/subscribe` — Register device for push:
  - Parse subscription object from service worker (endpoint, auth, p256dh)
  - Store subscription with user ID, device info
  - Return confirmation with subscription ID
- `POST /api/push/unsubscribe` — Unregister device:
  - Identify subscription by ID or endpoint
  - Remove from store
  - Return 204 No Content on success
- `GET /api/push/subscriptions` — List user's active subscriptions:
  - Return array of registered devices with metadata (created, last_used, device_name)
  - Allow filtering by device type or status
- `PUT /api/push/events` — Configure notification events:
  - Parse event configuration (which events trigger push: session_started, message_received, etc.)
  - Store user's event preferences
  - Return updated configuration
- `POST /api/push/answer` — Answer push notification prompt:
  - Handle user response from push notification (e.g., "run code" → true/false)
  - Route answer to appropriate handler
  - Return acknowledgment

**Dependencies:**
- `packages/server/src/push.ts` — Push notification logic
- `packages/server/src/auth.ts` — User identity
- VAPID key store (static config or database)

## Considerations

### Chat Router
- **External API dependency**: Models API is external; mocking is essential for CI portability
- **Tool calling**: May involve spawning runners; ensure proper error handling and timeouts
- **Streaming**: If chat responses are streamed, ensure proper chunking and error recovery
- **Rate limiting**: Consider rate limiting per user/model to prevent abuse

### Push Router
- **VAPID keys**: Must be generated during setup; store securely (not in code)
- **Subscription storage**: Should be durable (DB) with proper cleanup of stale subscriptions
- **Security**: VAPID signing keys must never be exposed to client
- **Service Worker protocol**: Ensure server-to-client push format matches service worker expectations
- **Subscription lifecycle**: Handle subscription invalidation when user logs out or browser clears data
