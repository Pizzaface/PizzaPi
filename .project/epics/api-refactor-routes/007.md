### Router Structure Documentation (`routes/README.md`)

```markdown
# API Routes

This directory contains the modular API router implementations for PizzaPi.

## Architecture

The API is organized into seven domain-specific routers:
- **auth.ts** — User authentication (register, signup status)
- **settings.ts** — User preferences (hidden models)
- **runners.ts** — Runner lifecycle (spawn, restart, stop, terminal, skills, files, git)
- **sessions.ts** — Session management (list, pinned, pin/unpin)
- **attachments.ts** — File attachments (upload, download)
- **chat.ts** — Chat/messaging (chat, models)
- **push.ts** — Push notifications (subscribe, events, VAPID)

## Router Pattern

All routers implement the same interface:

\`\`\`typescript
export async function handleXyzRoute(
  req: Request,
  url: URL
): Promise<Response | undefined>
\`\`\`

Routers:
- Return `Response` if the path matches
- Return `undefined` if the path doesn't match (dispatcher tries next router)
- Never throw uncaught errors (catch and return error Response)

## Dispatcher

The dispatcher in `handler.ts` calls routers sequentially:
1. Auth validation in handler.ts
2. Sequential path matching: /api/auth/* → auth router, etc.
3. Return 404 if no router matches

## Quick Reference

| Path | Router | Purpose |
|------|--------|---------|
| /api/auth/* | auth | User registration, signup status |
| /api/settings/* | settings | User preferences |
| /api/runners/* | runners | Runner management |
| /api/sessions/* | sessions | Session list, pinning |
| /api/attachments/* | attachments | File upload/download |
| /api/chat | chat | Chat messaging |
| /api/push/* | push | Push notifications |

## Adding a New Endpoint

See [CONTRIBUTING.md](CONTRIBUTING.md) for step-by-step guide.

## Testing

See [TESTING.md](TESTING.md) for testing patterns and best practices.
```

## Router Pattern Recap

```typescript
// Standard router signature
export async function handleXyzRoute(
  req: Request,
  url: URL
): Promise<Response | undefined> {
  const { method, headers } = req;
  const pathname = url.pathname;

  // Return undefined for paths not handled by this router
  if (!pathname.startsWith("/api/xyz/")) {
    return undefined;
  }

  // Handle specific endpoints
  if (method === "GET" && pathname === "/api/xyz/list") {
    // Your logic here
    return new Response(JSON.stringify({ /* ... */ }));
  }

  if (method === "POST" && pathname === "/api/xyz/create") {
    // Your logic here
    return new Response(JSON.stringify({ /* ... */ }), { status: 201 });
  }

  // Return undefined for unrecognized paths in this domain
  return undefined;
}
```

Key points:
- Always return `undefined` if path doesn't start with your domain
- Always return `Response` if path matches (even if it's an error)
- Never throw uncaught exceptions
- Use consistent error handling (400, 401, 403, 404, 500)

# Router Testing

## Test File Location

Each router has a co-located test file:
- `routes/auth.ts` → `routes/auth.test.ts`
- `routes/runners.ts` → `routes/runners.test.ts`

## Test Template

```typescript
import { describe, test, expect } from "bun:test";
import { handleXyzRoute } from "./xyz";

describe("Xyz Router", () => {
  test("GET /api/xyz returns data", async () => {
    const req = new Request("http://localhost/api/xyz", {
      method: "GET"
    });
    const url = new URL(req.url);

    const res = await handleXyzRoute(req, url);

    expect(res).toBeDefined();
    expect(res?.status).toBe(200);
    const data = await res?.json();
    expect(data).toHaveProperty("key");
  });

  test("returns undefined for unmapped paths", async () => {
    const req = new Request("http://localhost/api/other/path", {
      method: "GET"
    });
    const url = new URL(req.url);

    const res = await handleXyzRoute(req, url);

    expect(res).toBeUndefined();
  });

  test("handles errors gracefully", async () => {
    const req = new Request("http://localhost/api/xyz/invalid", {
      method: "POST",
      body: JSON.stringify({ /* invalid */ })
    });
    const url = new URL(req.url);

    const res = await handleXyzRoute(req, url);

    expect(res?.status).toBe(400); // or appropriate error code
  });
});
```

## Mocking Patterns

### Mock External Service
```typescript
const mockFetch = jest.fn();
global.fetch = mockFetch;
mockFetch.mockResolvedValueOnce(new Response(JSON.stringify({ /* ... */ })));
```

### Mock Database
```typescript
// Import and mock the DB module
import * as db from "../db";
db.query = async (sql) => {
  return [{ id: 1, name: "Test" }];
};
```

### Mock Auth Context
```typescript
// Pass user context via request headers
const req = new Request("http://localhost/api/protected", {
  method: "GET",
  headers: { "x-user-id": "user123" }
});
```

## Common Testing Pitfalls

❌ **Don't**: Call external APIs in tests
✓ **Do**: Mock HTTP requests/responses

❌ **Don't**: Depend on system state (files, env vars)
✓ **Do**: Use in-memory mocks or temp directories

❌ **Don't**: Test implementation details
✓ **Do**: Test observable behavior (request in → response out)

❌ **Don't**: Skip error cases
✓ **Do**: Test both success and failure paths
```
