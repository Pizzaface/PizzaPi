# Task: Extract sessions & attachments routers

## Description

Refactor session management and file attachment endpoints into dedicated routers. Sessions and attachments are logically related (attachments belong to sessions) but can be split into two routers for clarity. These routers handle session metadata (listing, pinning) and file lifecycle (upload, download), with careful attention to permission checks and file streaming.

**Endpoints:**
- `POST /api/sessions/:id/attachments` — Upload file:
  - Parse `multipart/form-data` or `application/octet-stream`
  - Validate file size (enforce limit, typically 100MB or configurable)
  - Store in attachment store with metadata (filename, size, MIME type)
  - Return attachment ID and metadata
- `GET /api/attachments/:id` — Download file:
  - Verify user has permission to access attachment
  - Stream file contents (don't buffer entire file in memory)
  - Set appropriate Content-Type, Content-Disposition headers
  - Return file with correct Content-Length

**Dependencies:**
- `packages/server/src/attachments/store.ts` — File attachment storage
- `packages/server/src/security.ts` — Permission checks

## Considerations

### Sessions Router
- Session listing should be efficient (may need pagination for users with many sessions)
- Pin/unpin operations should be idempotent (pinning twice = one pin)
- Pinned sessions could be user preference; verify storage mechanism

### Attachments Router
- **File upload**: Use streaming to avoid loading entire file in memory
- **Size limits**: Enforce reasonable limits (e.g., 100MB per file, 1GB per session)
- **Security**: Verify user has permission to attach to session (permission checking)
- **Streaming download**: Use `ReadableStream` or similar to avoid buffering large files
- **MIME types**: Set correct Content-Type based on file extension or upload metadata
