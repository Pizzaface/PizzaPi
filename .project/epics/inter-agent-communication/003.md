# Task: CLI Event-Driven Message Delivery via sendUserMessage

## Description

Implement automatic message injection for inter-agent communication. When a session receives a `session_completion` event or `session_message` from another agent, it is automatically injected into the agent's conversation via `pi.sendUserMessage()` rather than requiring the agent to poll with `wait_for_message` or `check_messages`. This is the highest-value single task — it unlocks the core UX improvement of the epic.

## Acceptance Criteria

- [ ] Incoming `session_completion` events are automatically formatted and injected via `pi.sendUserMessage()` when the agent is idle
- [ ] Incoming `session_message` events with agent metadata are handled based on delivery mode
- [ ] Three delivery modes implemented: `"immediate"` (inject now), `"queued"` (inject after current turn), `"blocked"` (manual poll only — backward-compatible default for existing `send_message`/`wait_for_message` flow)
- [ ] Default delivery mode: `"queued"` for worker sessions (prevents interruption mid-turn), `"immediate"` for idle sessions
- [ ] Completion messages are given higher priority than regular messages in the queue
- [ ] After each `agent_end`: drain queued messages and inject them as the next user turn
- [ ] Message formatting: `[AGENT_MESSAGE from={sessionId} type={completion|message}]\n{content}` prefix for agent distinguishability
- [ ] Rate-limiting: max 5 injections per minute to prevent infinite loops (per AD-1)
- [ ] Message injection adds < 50ms latency to turn start (performance criterion)
- [ ] Existing `send_message`/`wait_for_message`/`check_messages` tools continue to work unchanged
- [ ] All existing tests pass
- [ ] New unit tests for delivery modes, queue draining, rate-limiting, and message formatting

## Technical Details

### Session Message Bus Changes (`packages/cli/src/extensions/session-message-bus.ts`)

- Add delivery mode enum: `"immediate" | "queued" | "blocked"`
- Add completion message queue (separate from regular messages, higher priority)
- Add `setDeliveryMode(mode)` method
- Add `onMessageReady(callback)` — fires when a message should be injected (called by remote.ts)
- Add `drainQueue(): string[]` — returns all queued messages, clears queue
- Separate completion queue and regular message queue; drain completions first

### Remote Extension Changes (`packages/cli/src/extensions/remote.ts`)

- Listen for `session_completion` events from relay
- On `session_completion` received:
  1. Format as `[AGENT_MESSAGE from={sessionId} type=completion]\n{result}`
  2. If agent is idle → call `pi.sendUserMessage()` immediately
  3. If agent is busy → add to completion queue for post-turn injection
- On `session_message` with agent metadata:
  1. Check delivery mode
  2. `"immediate"`: inject now if idle, queue if busy
  3. `"queued"`: always queue, drain after current turn
  4. `"blocked"`: do nothing (leave for `wait_for_message` to pick up)
- After each `agent_end`: call `messageBus.drainQueue()`, inject combined messages as next user turn
- Rate-limiter: track injection timestamps, skip if > 5/minute (log warning)
