  1. **Race condition â€” `depends_on: - ui` uses short form (service_started, not service_healthy).**
     `packages/cli/src/web.ts:802`. The generated compose file writes:
     ```yaml
     depends_on:
       - redis
       - ui
     ```
     Docker Compose short form means `condition: service_started` â€” the server container is allowed to start as soon as the `ui` container process has been launched, _not_ after the healthcheck passes. The `ui` container's entrypoint runs `rm -rf /ui-dist/* && cp -a /usr/share/nginx/html/. /ui-dist/ && tail -f /dev/null`. On any system where that copy takes more than a few milliseconds (cold image pull, slow volume I/O, arm emulation), the server can attempt to serve the mount path before `index.html` exists. The healthcheck (`test: ["CMD", "test", "-f", "/ui-dist/index.html"]`) was clearly written to guard against this, but it is only useful when `depends_on` uses `condition: service_healthy`.
     **Fix:** Convert to long-form depends_on:
     ```yaml
     depends_on:
       - redis
       ui:
         condition: service_healthy
     ```
     Note: short (`- redis`) and long-form entries cannot be mixed in the same block; redis should also be moved to long form with `condition: service_started`.

  2. **`packages/ui/.dockerignore` is placed at the wrong path and is silently ignored.**
     `packages/ui/.dockerignore`. Docker reads `.dockerignore` from the _build context root_ (`.dockerignore`) or from a companion file named `<dockerfile-path>.dockerignore` (`packages/ui/Dockerfile.dockerignore`). Neither of those is `packages/ui/.dockerignore`, so the file has no effect. The root `.dockerignore` (which is used) excludes `node_modules` and `*/node_modules` but does _not_ exclude `packages/ui/dist`, `packages/ui/dev-dist`, or `packages/ui/coverage`. As a result any host-built UI dist (or coverage output) is included in the build context sent to the Docker daemon. For `packages/ui/dist` this is harmless because the Dockerfile rebuilds it, but `dev-dist` and `coverage` are dead weight on every build and could be large. More importantly, the miss means the intent of the author is not enforced.
     **Fix:** Either rename to `packages/ui/Dockerfile.dockerignore` (companion form), or add the missing patterns to the existing root `.dockerignore`.

- **P3:**
  6. **`nginx:alpine` base image not digest-pinned in the UI Dockerfile.**
     `packages/ui/Dockerfile:37`. Using a floating `:alpine` tag is fine for typical CI but means the base image can change silently between builds. For a published GHCR image that users pull as a trusted artefact, a pinned digest (`nginx:alpine@sha256:â€¦`) provides a stronger supply-chain guarantee.

  7. **Named volume `ui-dist` persists after `pizza web stop`.**
     `packages/cli/src/templates/compose.yml.template`. `docker compose down` removes containers but not named volumes unless `--volumes` is passed. Users who run `pizza web` again after a failed or interrupted pull will be served whatever was left in the `ui-dist` volume from the previous run rather than seeing an error. This is especially surprising when `--tag` changes. Consider documenting this, or having `pizza web` prune the volume before pulling the new image.

  8. **No multi-platform build in CI workflow.**
     `.github/workflows/ui-docker.yml`. The `build-push-action` defaults to the runner architecture (`linux/amd64`). Users deploying on `linux/arm64` (Raspberry Pi, Ampere servers, etc.) will receive an amd64 image and rely on emulation or experience a pull failure depending on their Docker daemon configuration. Adding `platforms: linux/amd64,linux/arm64` with `docker/setup-qemu-action` would cover the common ARM case.

### Verdict

**CITATION**

The PR is well-structured and the three features are logically sound. Test coverage is good; protocol version negotiation is correctly backwards-compatible (missing `protocolVersion` treated as compatible). The version-banner UI and migration-check logging are clean additions.

Two P1 issues must be resolved before the GHCR mode is reliable in production:

1. The `depends_on` race (P1-1) means the server can start serving a blank UI mount on any slightly-loaded system. The fix is a single-line change to `uiDependsOnLine` in `web.ts:802`.
2. The mis-placed `.dockerignore` (P1-2) means the intent of keeping the GHCR build context clean is not enforced. Easy fix: rename or move the file.

The P2s (triple `UI_VERSION` duplication, dual mismatch notification path, missing tsconfig in workflow triggers) should be addressed but are not blockers.
