### Detail

#### Fix 1 — Tunnel relay timeout resource leak (`packages/tunnel/src/server.ts`)

**What changed:** Two lines added — one before `pendingRequests.delete` in the HTTP timeout callback (L139), one before `pendingWs.delete` in the WS open-timeout callback (L215):

```ts
// HTTP (line ~139)
this.send(runner.ws, { type: "request-end", id: request.id });
this.pendingRequests.delete(request.id);
callbacks.onError("Tunnel request timed out");

// WS (line ~215)
this.send(runner.ws, { type: "ws-close", id: request.id, code: 1001, reason: "open timeout" });
this.pendingWs.delete(request.id);
callbacks.onError("WebSocket open timed out");
```

**Race analysis — clean:**  
JS is single-threaded. The timer fires once; `send()` guards on `ws.readyState === WebSocket.OPEN` so a disconnected runner is a safe no-op. `handleDisconnect` already calls `clearTimeout(pending.timer)` before it deletes pending entries, so the timeout can never fire after a runner disconnect.

If a `response-start` / `response-data` / `response-data-end` arrives from the runner after the timeout has fired and deleted the entry, every `handleResponse*` handler does `if (!pending) return` — no double-callback, no corruption.

WS variant: `handleWsOpened` clears the timer when the open succeeds, so the new `ws-close` on timeout can only fire if the WS never opened within the window. Even if `ws-opened` races in right after the relay fires `ws-close`, `handleWsOpened` gets `pending = undefined` and returns early. Safe.

Runner-side handling is correct in both cases:
- `handleRequestEnd` aborts and destroys the in-flight `http.ClientRequest`; if the request already completed, `activeRequests.get` returns `undefined` and it's a no-op.
- `handleWsClose` closes the local WebSocket regardless of its ready-state; the subsequent `close` event on the runner sends a `ws-close` back to the relay, which the relay discards because `pendingWs` was already cleared.

**P2 — No unit test for the timeout cancellation path:**  
`packages/tunnel/src/server.test.ts` has nine tests covering registration, response callbacks, and disconnect. None of them use fake timers to assert that `request-end` or `ws-close` is sent when the timeout fires. The two added lines — the entire fix — are completely untested. The existing mock infrastructure (`createMockWebSocket`, `waitForMicrotask`) is sufficient to write such a test with `bun:test`'s `setSystemTime` / `advanceTimers`. This should be added.

#### Fix 2 — Server graceful shutdown (`packages/server/src/index.ts`)

**What changed:**
- `uncaughtException` handler expanded: EPIPE already swallowed; new: ECONNRESET, ERR\_STREAM\_DESTROYED, ENOTCONN swallowed at `warn` level; `SyntaxError` with "json" in message swallowed at `warn` level; anything else calls `onShutdownSignal("uncaughtException", 1)` inside a `try/catch` (fall-through to `process.exit(1)` if server not yet ready).
- `onShutdownSignal` gains: `exitCode` param (default 0), re-entrancy guard (`let shuttingDown = false`), `closeIdleConnections?.()` for faster drain.
- `process.exit(0)` inside the drain callbacks changed to `process.exit(exitCode)`.

**Shutdown paths — clean:**  
SIGTERM/SIGINT → `onShutdownSignal(signal, 0)` → drain → `process.exit(0)`. Force-exit after 10 s with `process.exit(1)`. The `io` branch and the no-Socket.IO branch are both updated consistently. `closeIdleConnections?.()` correctly uses optional chaining for Bun compatibility.

**Re-entrancy guard — clean:**  
`let shuttingDown = false` guards against SIGTERM + SIGINT double-trigger and SIGTERM + transient exception during drain.

**Hoisting claim — accurate:**  
`function onShutdownSignal` is a `function` declaration at module scope; ES modules hoist function declarations, so it is callable from the `uncaughtException` handler registered at line 46 even though its source position is line ~422.

**P2 — Exit code corruption when a fatal exception fires during the drain window (`packages/server/src/index.ts` ~L78, L422-L460):**

Scenario:
1. SIGTERM arrives → `onShutdownSignal("SIGTERM", 0)` sets `shuttingDown = true`, begins drain, schedules `process.exit(0)` inside `io.close(→ closeHttp())` callbacks.
2. Before drain completes, a fatal exception (not in the recoverable list) fires.
3. `uncaughtException` handler calls `onShutdownSignal("uncaughtException", 1)`.
4. Guard sees `shuttingDown === true` → logs a warning and **returns early, discarding `exitCode = 1`**.
5. Drain completes normally → **`process.exit(0)`** — wrong exit code.

The error IS logged at `processLog.error`, so it's visible in log pipelines. The exit code being 0 means process managers and Kubernetes liveness/restart policies based on exit code will not register a crash. The 10-second force-exit with `process.exit(1)` only fires if the drain stalls — it does not help when the drain completes within the window.

Minimal fix — track the worst exit code seen:
```ts
let pendingExitCode = 0;   // module-level, alongside shuttingDown

function onShutdownSignal(signal: string, exitCode = 0): void {
    if (exitCode > pendingExitCode) pendingExitCode = exitCode;   // escalate, never de-escalate
    if (shuttingDown) {
        shutdownLog.warn(`Received ${signal} during shutdown — ignoring duplicate signal`);
        return;
    }
    // ...
    closeHttp(() => {
        process.exit(pendingExitCode);   // uses highest-severity code seen
    });
}
```

**P3a — TDZ window comment incomplete (`packages/server/src/index.ts` ~L72-L83):**

The comment reads *"Server not yet fully initialized (e.g. exception during migrations). Fall back to immediate exit — there is nothing to drain."* It attributes the `try/catch` to `httpServer` not existing yet. In practice, `shuttingDown` (declared at line 420) is equally in the TDZ for any exception between lines 46–420 (e.g., during migrations at line 114 or during Redis connection setup). The first thing `onShutdownSignal` does is `if (shuttingDown)` — accessing `shuttingDown` in TDZ throws `ReferenceError` before `httpServer` is even reached. Both are caught by the try/catch; behavior is correct. The comment should mention both bindings.

---

**P3b — ECONNRESET / ERR\_STREAM\_DESTROYED / ENOTCONN swallowing (`packages/server/src/index.ts` ~L56-L63):**

These codes are connection-scoped and the server state is genuinely unaffected, so `warn`+continue is defensible. However, reaching `uncaughtException` means a `try/catch` was missed in the call graph. In the original code these would crash the server (surfacing the missing boundary). Silently swallowing them means a latent error boundary gap will never be found unless someone actively monitors `warn` logs. No correctness bug, but worth noting as a deliberate observability trade-off.

#### Fix 3 — Service panel auto-focus bug (`packages/ui/src/App.tsx`)

**Adding bug fix (L3544-L3557):**
```tsx
if (activeServicePanels.has(combinedActiveTab)) {
    setServicePanelPosition(serviceId, getServicePanelPosition(combinedActiveTab));
}
toggleServicePanel(serviceId);
handleCombinedTabChange(serviceId);
```

When a second service panel is opened while another service panel is the active tab, the new panel is moved to the same position group before being toggled open. Because React 19 batches all three state updates (`setPanelPositions`, `setActivePanelIds`, `setCombinedActiveTab`) into a single re-render, the `panelGroups` memo sees the correct position on the very first render. No stale read issue.

**Moving bug fix (L3639-L3648):**
```tsx
onDragStart: (e) => startPanelDragWith(e, (pos) => {
    setServicePanelPosition(serviceId, pos);
    handleCombinedTabChange(serviceId);   // ← new
}),
```

Re-asserts focus on the dragged panel after its position changes, preventing the destination group from falling back to `tabs[0]` (Tunnels). Correct.

**Dependency arrays:** Both `handleToggleServicePanel` and the `servicePanelTabs` memo have their dependency arrays updated to include the newly-used identifiers (`combinedActiveTab`, `setServicePanelPosition`, `getServicePanelPosition`, `handleCombinedTabChange`). No stale closure risk.

**`getPanelPosition` stability:** `getServicePanelPosition` is a `useCallback` in `useServicePanelState` that depends on `[panelPositions]`. Adding it to a dependency array is correct — it will change reference only when `panelPositions` Map changes (on a position update), which is exactly when the dependent memo needs to recompute.

**Test coverage:** `ServicePanels.test.ts` (206 lines, 11 tests) tests the pure decision logic for both sub-bugs. Tests correctly cover the adding case, the no-active-panel case, the stale-stored-position override, and the moving case. All pass. Coverage of the App component wiring itself is absent (React component testing boundary), which is acceptable and consistent with how `usePanelLayout.test.ts` is structured.

#### `App.tsx` session-state consolidation (ancillary to Fix 3)

**What changed:** ~30 `useState` calls collapsed into a single `SessionState` object with `setSessionState`. `clearSelection` now calls `setSessionState(createInitialSessionState())` atomically.

**Correctness:** All individual setters use the functional-update pattern `(p) => ({ ...p, field: ... })`, which is correct for concurrent-mode React. The `typeof v === "function"` guard correctly distinguishes functional updaters from direct values; none of the `SessionState` fields are function-typed, so the only practical path is the value branch.

**Intentional exclusion:** `isCompacting` and `planModeEnabled` remain as standalone `useState` — they survive session switches by design. Comment documents this. Correct.

**P3c — Setter boilerplate volume:** 30+ identical `useCallback` wrappers (L~220-L~380 in the PR) are verbose. A `useReducer` with a discriminated-union action type would be the idiomatic React pattern and would eliminate the boilerplate. No functional problem with the current approach, and it preserves backward-compatible call-site signatures, but maintainability cost is worth noting for a future refactor.
