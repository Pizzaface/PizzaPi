# Inspection: PR #363 — consistent error messaging

## Summary

Introduces `packages/ui/src/lib/user-error-message.ts` — a central `mapUserError()` function that translates raw errors into user-friendly strings. Applies it across viewer socket error events, spawn flows in `App.tsx`, and runner restart/stop flows in `RunnerManager.tsx`. Ships a test file with 6 cases.

Direction is correct. Two P1 issues prevent a clean bill.

#### P1-A · Server error messages silently swallowed by generic fallback

**File:** `packages/ui/src/App.tsx` ~line 2568 (branch) / `packages/server/src/ws/namespaces/viewer.ts` lines 151, 167, 186

The viewer socket's `"error"` event handler now routes everything through `mapUserError`. The server sends three distinct messages from `viewer.ts`:

| Server message | Old status shown | New status shown |
|---|---|---|
| `"Session not found"` | `"Session not found"` | `"Failed to load session."` |
| `"Session snapshot not available"` | `"Session snapshot not available"` | `"Failed to load session."` |
| `"Failed to load session snapshot"` | `"Failed to load session snapshot"` | `"Failed to load session."` |

None of these strings match any keyword check in `mapUserError`, so they all fall through to `input.fallbackMessage` ("Failed to load session."). The result is correct for the third case — the fallback is equivalent — but wrong for the first two:

- **"Session not found"** → user should understand the session is gone (pruned, never existed). The fallback "Failed to load session." implies a transient error and will prompt unnecessary retries. This is a meaningful UX regression for the most common non-live session state.
- **"Session snapshot not available"** → similar — cache/persistence miss, different cause than a network failure.

The fix is to add keyword checks for the server's actual strings, e.g.:

```ts
if (normalized.includes("session not found")) {
    return {
        userMessage: "This session no longer exists. It may have ended or been removed.",
        technicalMessage,
    };
}
if (normalized.includes("snapshot not available") || normalized.includes("load session snapshot")) {
    return {
        userMessage: "Session history couldn't be loaded. Try refreshing.",
        technicalMessage,
    };
}
```

#### P1-B · Test for `connect_error` covers fabricated input that doesn't match production

**File:** `packages/ui/src/lib/user-error-message.test.ts` lines 43–48

```ts
test("maps viewer connection network failures", () => {
    const result = mapUserError({
        error: "connect_error: xhr poll error",   // ← fabricated string
        context: "viewer_connection",
    });
    expect(result.userMessage).toContain("Lost connection to PizzaPi");
});
```

The test passes because `"connect_error: xhr poll error"` contains the substring `"connect_error"`. But socket.io's actual `connect_error` event delivers an `Error` **object** whose `.message` is `"xhr poll error"` — no prefix. `coerceErrorMessage(err)` returns `"xhr poll error"`, which doesn't match `"connect_error"`, `"websocket"`, `"failed to fetch"`, or `"network error"`:

```
"xhr poll error" -> NO MATCH  (verified against installed socket.io-client@4.8.3)
"websocket error" -> MATCHED
```

In production the error falls through to the context fallback `"Couldn't connect to the session. Check your connection and try again."` — not `"Lost connection to PizzaPi"`. The test gives false confidence that the specific message fires for the most common socket.io transport failure.

Fix: pass an actual `Error` object and/or add `"xhr poll error"` / `"transport error"` to the keyword list. The fallback is still acceptable UX, but the test should reflect what really happens:

```ts
test("maps viewer connection xhr poll error (real socket.io format)", () => {
    const result = mapUserError({
        error: new Error("xhr poll error"),
        context: "viewer_connection",
    });
    // This currently falls through to the context fallback — test that
    expect(result.userMessage).toContain("Check your connection");
});
```

#### P2-A · Auth error copy over-promises — no sign-in action is available in viewer context

**File:** `packages/ui/src/lib/user-error-message.ts` lines 90–95

```ts
if (normalized.includes("forbidden") || normalized.includes("unauthorized") || statusCode === 401 || statusCode === 403) {
    return {
        userMessage: "You don't have access to do that right now. Sign in again and retry.",
        technicalMessage,
    };
}
```

This message appears in `viewerStatus` (bottom status bar). When session auth expires and a 401/403 is returned, the status bar says "Sign in again and retry" — but there is no sign-in button in the viewer, and no auto-redirect happens. The user must know to reload the page entirely.

Suggested copy: `"Session authentication expired. Reload the page to sign in again."`

#### P2-B · Generic `>=500` handler preempts context-specific fallbacks

**File:** `packages/ui/src/lib/user-error-message.ts` lines 119–124

```ts
if (statusCode !== undefined && statusCode >= 500) {
    return {
        userMessage: "PizzaPi hit a temporary server problem. Please retry in a moment.",
        technicalMessage,
    };
}
```

This fires unconditionally, so a `runner_stop` or `runner_restart` that returns HTTP 500 shows `"PizzaPi hit a temporary server problem."` instead of `"Couldn't stop the runner. Please try again."` The context-specific message is more actionable (names the operation). Similarly, a `session_spawn` 500 bypasses the context fallback.

The generic 500 handler should either be eliminated in favor of the context fallback, or placed after the fallback returns (i.e., only apply when context is `"generic"`).

#### P3-A · Structured logger removed, replaced with `console.error`

**File:** `packages/ui/src/components/RunnerManager.tsx`

```diff
-import { createLogger } from "@pizzapi/tools";
-const log = createLogger("runner-ui");
+import { mapUserError } from "@/lib/user-error-message";
...
-log.error("Failed to restart runner:", err);
+console.error("Failed to restart runner:", err);
```

`createLogger` was only used for two calls, but their removal is a minor regression: structured log output (prefixed, filterable) is downgraded to raw `console.error`. Other components in the project still use `createLogger`. If the logger is intentionally being dropped here that's fine, but it should be a deliberate choice noted in the PR.
