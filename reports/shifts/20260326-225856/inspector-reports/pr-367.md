#### P2-A тАФ `service_announce` broadcast starvation after runner crash + reconnect
**File:** `packages/server/src/ws/namespaces/runner.ts:705тАУ708`

When a runner crashes (non-graceful disconnect), `runnerServiceAnnounce.delete(runnerId)` is called, clearing the in-memory cache. If a viewer then connects to any session on that runner, `viewer.ts:570` calls `getRunnerServiceAnnounce()`, finds an empty cache, and sends **no** `service_announce` тАФ so the viewer enters with no service panels rendered.

When the runner reconnects, `seedServiceAnnounceCache` re-seeds the cache from Redis with the same payload that was persisted before the crash. The new dedup gate (`isSameServiceAnnounce(previous, data)`) then compares the freshly-seeded value against the runner's re-announce and concludes "no change" тАФ returning `true` and **skipping the broadcast entirely**.

Result: the viewer that joined during the offline gap never receives a `service_announce` and service panels remain unavailable until the user manually reconnects the viewer socket.

**Pre-PR behaviour:** the broadcast always fired on runner reconnect (no dedup), so all in-flight viewers received the announce regardless of when they connected.

**Narrow window, but real regression.** The practical severity is low тАФ the gap is typically sub-second and the workaround is a viewer refresh тАФ but it is a silent functional regression.

**Suggested fix:** reset the guard for the reconnect-announce case by not calling `seedServiceAnnounceCache` before comparing, or by tracking a "needs broadcast" flag set when the cache was cleared on disconnect. The simplest safe fix is to clear `previous` to `null` after the cache is deleted so the first post-reconnect announce always broadcasts:

```ts
// disconnect handler (already in runner.ts ~L745)
runnerServiceAnnounce.delete(runnerId);
// ...existing cleanup...
```

Then in the `service_announce` handler, the `runnerServiceAnnounce.get(runnerId)` call returns `undefined` (since `seedServiceAnnounceCache` was not yet called), making `isSameServiceAnnounce(undefined, data)` return `false` and the broadcast fires. **The issue is specifically that `seedServiceAnnounceCache` runs during `register` (L372) before the announce arrives, pre-populating the cache and making the dedup think nothing changed.** Fix: skip the dedup when `seedServiceAnnounceCache` was the source of `previous` (e.g., add a `seededFromRedis` flag), or broadcast unconditionally on the first announce after a fresh connection.

#### P2-B тАФ 1200ms stale state in the "already connected" sidebar path
**File:** `packages/ui/src/components/SessionSidebar.tsx:761тАУ764`

The old code on the `hubSocket.connected` fast-path:
```ts
void resyncLiveSessionsFromApi(epoch);   // immediate
```

The PR changes this to:
```ts
scheduleResyncFallback(epoch);           // 1200ms delay
```

The accompanying comment was updated from "we **will** have missed the server's initial 'sessions' emit" (original тАФ certain) to "we **may** have missed" (PR тАФ uncertain). The certainty was correct. The `/hub` namespace emits `sessions` exactly once, at namespace join. If the socket was already connected before the sidebar's `useEffect` registered `handleSessions`, that event is gone and no new one will arrive. The 1200ms timer **always** fires in this path тАФ it is not a fallback, it is the only path.

Observable impact: session list shows blank/stale state for ~1.2s every time:
- The sidebar remounts with an already-live socket (route changes, conditional rendering).
- React Strict Mode double-invokes effects in dev (second invocation sees `hubSocket.connected = true`).

**Pre-PR:** REST fetch was immediate, session list populated in ~50ms on localhost.

**Fix:** restore the fast path for "already connected" and use the delayed fallback only for fresh `connect` events:

```ts
if (hubSocket.connected) {
    setDotState("connected");
    const epoch = ++connectEpoch;
    // Socket was connected before our listener registered; sessions event
    // already fired and was missed тАФ fall back immediately.
    void resyncLiveSessionsFromApi(epoch);
}
```

#### P3-A тАФ `isSameServiceAnnounce` is order-sensitive
**File:** `packages/server/src/ws/namespaces/runner.ts:239тАУ244`

The comparator does an ordered index-by-index scan of `serviceIds`. If the runner ever emits `["inspector","tunnel"]` when the cache holds `["tunnel","inspector"]`, the comparator returns `false` and triggers a broadcast even though the set of services is identical. This is a **false negative** (conservative direction тАФ it over-broadcasts, never under-broadcasts), so there is no correctness risk. Noted for completeness; fix only warranted if runners are observed to reorder their service ID arrays.

#### P3-B тАФ No automated coverage for `scheduleResyncFallback` timing
**File:** `packages/ui/src/components/SessionSidebar.tsx`

The new timer/cancellation logic (`scheduleResyncFallback`, `clearFallbackTimer`, epoch guard in timer callback) has no corresponding test. Given the subtle race it is designed to prevent, a test using fake timers (`jest.useFakeTimers` / Bun's timer mocking) that exercises the "sessions arrives before timer fires тЖТ timer cancelled" and "sessions never arrives тЖТ REST fetch called after 1200ms" cases would be valuable. The AGENTS.md testing standard says "all new code must include tests."
