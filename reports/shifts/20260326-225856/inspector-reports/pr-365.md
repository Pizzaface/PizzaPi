### P1 — Out-of-order finalization breaks `dedupedForSideEffects` invariant

**File:** `packages/ui/src/App.tsx`, `session_messages_chunk` handler (~line 1428–1510)

The PR changes the finalization gate from the local `isFinal` flag to the independent `canFinalizeChunkHydration()` test. This breaks a critical invariant.

**Old code** (pre-PR):
```
let dedupedForSideEffects = null;
setMessages((prev) => {
  if (!isFinal) return appended;      // only set when isFinal=true
  dedupedForSideEffects = deduped;
  return deduped;
});

if (isFinal) {                        // same flag — invariant: dedupedForSideEffects always non-null here
  if (dedupedForSideEffects) {
    setActiveToolCalls(...);
    patchSessionCache(...);
  }
}
```

**New code** (post-PR):
```
let dedupedForSideEffects = null;
setMessages((prev) => {
  if (!isFinal) return appended;      // only set when evt.final=true
  dedupedForSideEffects = deduped;
  return deduped;
});

const readyToFinalize = canFinalizeChunkHydration(...);  // CAN be true when isFinal=false
if (readyToFinalize) {
  if (dedupedForSideEffects) {        // NULL in the out-of-order case!
    setActiveToolCalls(...);
    patchSessionCache(...);
  }
}
```

**Failure scenario (out-of-order delivery):**
1. Chunk 2 (final, `evt.final=true`) arrives first — `isFinal=true` → updater runs dedup (on partial data), `dedupedForSideEffects` is set. `readyToFinalize=false` (chunks 0 and 1 missing) → nothing finalizes.
2. Chunk 1 arrives last (`evt.final=false`) — `isFinal=false` → updater appends without dedup, `dedupedForSideEffects` remains `null` for this invocation. `readyToFinalize=true` → enters finalization block, but `if (dedupedForSideEffects)` is false.

Consequences:
- `patchSessionCache({ messages: … })` is never called → the session UI cache does not reflect the assembled, deduplicated message list. Tab-switching immediately after hydration shows stale/incomplete messages.
- `setActiveToolCalls(detectInFlightTools(…))` is never called → in-flight tool indicators and Kill buttons are not restored after a chunked reconnect.
- `sessionHydratedRef.current` is set to `true` and `chunkedDeliveryRef.current` is set to `null` regardless — so the session is marked "done" but the cache is wrong.

The PR adds out-of-order tests on the server side (`event-pipeline.test.ts`) and the UI helper layer (`session-seq.test.ts`) but the integration path in App.tsx is not tested end-to-end for this scenario, allowing the regression to go undetected.

**Minimal fix:** when `readyToFinalize` is true but `dedupedForSideEffects` is null (finalization triggered by a non-final chunk), fall back to deduplicating `messagesRef.current` instead of skipping the side effects:

```ts
if (readyToFinalize) {
  ...
  const finalMessages = dedupedForSideEffects ?? deduplicateMessages(messagesRef.current);
  if (finalMessages.length > 0) {
    // Also need to call setMessages with the deduped version if dedup didn't run in the updater
    if (!dedupedForSideEffects) setMessages(finalMessages);
    setActiveToolCalls(detectInFlightTools(finalMessages));
    patchSessionCache({ messages: finalMessages });
  }
}
```

### P2a — `dedupedForSideEffects` capture relies on synchronous updater execution

**File:** `packages/ui/src/App.tsx` ~line 1428–1461  
**Comment in code:** *"the updater above assigned `dedupedForSideEffects` synchronously before returning"*

In React 18 with `createRoot` (concurrent mode), `setState(updater)` queues the update — the updater function runs during the next render/commit phase, not inline at the callsite. The `dedupedForSideEffects` variable is declared as a local `let` and mutated inside the updater, then immediately read on the next line. This only works reliably if React calls the updater synchronously before the outer code continues.

While React 18 may flush the batch before the next browser task (making this work empirically in most environments), the pattern violates the React model that functional updaters must be pure and side-effect-free. Concurrent mode's speculative re-rendering can call updaters multiple times (the PR's own rationale for moving `patchSessionCache` _out_ of updaters elsewhere). The comment incorrectly asserts synchronous guarantees.

This is a **pre-existing pattern** the PR did not introduce, but the PR extends its reach. The right fix is to eliminate the mutation-from-inside-updater pattern entirely — compute the deduped list synchronously outside the updater (using `messagesRef.current`), then call `setMessages(deduped)` directly.

### P2b — Cross-chunk dedup skipped in out-of-order case

**File:** `packages/ui/src/App.tsx` ~line 1484–1488

The `setMessages` updater only runs `deduplicateMessages` when `isFinal` (from `evt.final`) is true. In the out-of-order case described in P1:
- When chunk 2 (final) arrives first, dedup runs on an **incomplete** message set — it deduplicates only the messages received so far.
- When the remaining non-final chunks arrive, they are appended without dedup.
- Because `readyToFinalize` fires with `dedupedForSideEffects=null`, the final full-set dedup is silently skipped.

Cross-chunk duplicates (e.g., a partial message in one chunk and its completed version in a later chunk) can persist in the rendered message list.

### Server-side (`event-pipeline.ts`) — CLEAN

`applyChunkToPendingState`, `hasAllChunkIndexes`, `canFinalizeChunkedSnapshot` are all correct:
- `finalChunkSeen` and `totalChunks` are updated before the duplicate-index guard, so a retransmit of the final chunk still sets both fields even though the data itself is deduplicated.
- `hasAllChunkIndexes` performs O(N) contiguous-coverage check — no counter increment that could be fooled by duplicate deliveries.
- `totalChunks` is only updated when positive, correctly refusing to overwrite a known total with a "not-yet-known" zero from a non-final retransmit.
- Tests cover: duplicate retransmit, gap scenario, final-chunk-only gate, out-of-order assembly. All four cases are correct.
