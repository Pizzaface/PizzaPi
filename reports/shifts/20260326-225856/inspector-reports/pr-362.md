#### P1 — MCP subprocess orphaned on dispose-during-connect

**`packages/cli/src/runner/services/godmother-service.ts` lines 310–332 (`ensureClient`) and 285–305 (`dispose`)**

`ensureClient()` stores the in-flight connect promise in `this.clientPromise` and, in the `.then()` callback, assigns the resolved client to `this.client`:

```ts
this.clientPromise = this.createClient(config)
    .then(async (client) => {
        await client.initialize();
        this.client = client;   // ← runs unconditionally after the await
        return client;
    })
    .catch((err) => {
        this.clientPromise = null;
        throw err;
    });
```

`dispose()` clears both fields synchronously:

```ts
this.client = null;
this.clientPromise = null;
```

If `dispose()` fires while `clientPromise` is in-flight (e.g., the socket drops during the first incoming request), the `.then()` callback still runs after dispose completes and writes a fresh `McpClient` reference back into `this.client`. At that point no owner will ever call `.close()` on it. The spawned stdio subprocess is permanently orphaned.

On flaky connections or rapid reconnects this is repeatable — each reconnect that triggers an MCP call mid-flight leaks a child process.

**Fix:** add a `disposed` guard checked inside `.then()` and close/discard the client if the service is already gone:

```ts
.then(async (client) => {
    await client.initialize();
    if (this.isDisposed) {
        try { client.close(); } catch { /* ignore */ }
        throw new Error("GodmotherService disposed during client init");
    }
    this.client = client;
    return client;
})
```

#### P2a — `list_ideas` never passes `limit` to the MCP tool

**`godmother-service.ts` line ~350 (`handleIdeasQuery`)**

When no search `query` is set the code calls `list_ideas` without the `limit` argument:

```ts
await this.callToolJson("list_ideas", {
    project,
    ...(status ? { status } : {}),
    ...(topic ? { topics: [topic] } : {}),
    include_completed: includeCompleted,
    // limit is not here
});
```

`limit` is computed (clamped to [1..200], default 60) and correctly forwarded to `search_ideas`, but is silently dropped for the browse path. The `slice(0, limit)` at line ~366 only applies after all items have already been fetched, normalized, and sorted in memory. If the Godmother MCP server has no internal default cap this can pull thousands of records per panel refresh.

**Fix:** add `limit` to the `list_ideas` call (the MCP API likely accepts it):

```ts
await this.callToolJson("list_ideas", {
    project,
    ...(status ? { status } : {}),
    ...(topic ? { topics: [topic] } : {}),
    include_completed: includeCompleted,
    limit,
});
```

#### P2b — `search_ideas` silently drops the `topic` filter

**`godmother-service.ts` line ~344 (`handleIdeasQuery`)**

When a `query` is present the `topic` parameter is NOT forwarded to `search_ideas`:

```ts
await this.callToolJson("search_ideas", {
    query,
    project,
    ...(status ? { status } : {}),
    limit,
    // topic is absent
});
```

The client-side filter `ideas.filter((idea) => idea.topics.some(...))` runs *after* the server has already truncated the result set to `limit` ranked results. Any idea whose topics match but that ranked below the limit will be silently excluded. Users who set both a search query and a topic filter will see fewer results than expected with no indication of why.

**Fix:** forward `topic` to `search_ideas` (or document that combined query+topic is unsupported):

```ts
await this.callToolJson("search_ideas", {
    query,
    project,
    ...(status ? { status } : {}),
    ...(topic ? { topic } : {}),
    limit,
});
```

#### P3a — `sessionId` prop accepted but never used

**`GodmotherPanel.tsx` line 147**

```tsx
export function GodmotherPanel(_props: GodmotherPanelProps) {
```

The `ServicePanelDef` contract (`registry.tsx`) specifies `component: React.ComponentType<{ sessionId: string }>` and `ServicePanelContainer` passes the active session's `sessionId` down. `TunnelPanel` uses it. `GodmotherPanel` silently discards it — the query payload sends a hardcoded `project: "PizzaPi"` rather than anything session-scoped.

If Godmother is intentionally global (not session-scoped) this is fine, but it should be documented. If session-scoping is ever desired the prop needs to be wired through `queryPayload` and into the service handler.

#### P3b — `topicDrafts` map grows unbounded across query refreshes

**`GodmotherPanel.tsx` lines 163, 307**

```ts
const [topicDrafts, setTopicDrafts] = useState<Record<string, string>>({});
```

Each user keystroke adds an entry keyed by `idea.id`. When `setIdeas(parseIdeaList(payload))` replaces the list on every query refresh, stale idea IDs are never pruned from `topicDrafts`. With 80 ideas per refresh and frequent queries the map accumulates unboundedly. Minimal footprint per entry, but the state never shrinks for the panel's lifetime.

**Fix:** prune stale keys when ideas are replaced:

```ts
setIdeas((parsedIdeas) => {
    const liveIds = new Set(parsedIdeas.map(i => i.id));
    setTopicDrafts(prev => {
        const next: Record<string, string> = {};
        for (const id of liveIds) if (id in prev) next[id] = prev[id]!;
        return next;
    });
    return parsedIdeas;
});
```

#### P3c — `idea_move_status` has no service-layer test

**`godmother-service.test.ts`**

Two of the three message handlers are tested (`ideas_query` and `idea_add_topics`). `idea_move_status` — including the `IDEA_STATUSES` allowlist check and the post-move `get_idea` re-fetch — has no coverage. This is the write path that mutates external state; it should be exercised.

#### P3d — `DEFAULT_PROJECT` hardcoded as `"PizzaPi"`

**`godmother-service.ts` line 55 / `GodmotherPanel.tsx` line 247**

```ts
const DEFAULT_PROJECT = "PizzaPi";
// …
project: "PizzaPi",
```

Both the server-side default and the UI's hardcoded query payload are pinned to `"PizzaPi"`. A user running PizzaPi against a different Godmother project will see no results without knowing why. Even a configurable default (read from the same `~/.pizzapi/config.json` that holds the MCP config) would be an improvement; exposing it as a UI dropdown would be better still.
