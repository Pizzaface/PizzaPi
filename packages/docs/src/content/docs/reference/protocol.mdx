## Overview

The `@pizzapi/protocol` package (`packages/protocol`) contains the shared TypeScript type definitions that every PizzaPi component ‚Äî CLI runner, relay server, and web UI ‚Äî uses to communicate over Socket.IO. It has **zero runtime dependencies** (dev-only `typescript`) and exports only types, constants, and small pure-function helpers.

By keeping wire types in a single package, any breaking change to the protocol surfaces as a compile error everywhere it matters, before anything ships.

## Socket.IO Namespaces

PizzaPi uses **six Socket.IO namespaces**, each with strongly-typed event maps for client‚Üíserver, server‚Üíclient, inter-server (Redis adapter), and per-socket metadata. Every namespace interface follows the same four-interface pattern:

| Interface suffix | Direction | Purpose |
|---|---|---|
| `ClientToServerEvents` | Client ‚Üí Server | Events the client emits |
| `ServerToClientEvents` | Server ‚Üí Client | Events the server emits |
| `InterServerEvents` | Server ‚Üí Server | Reserved for Redis adapter cross-node communication |
| `SocketData` | ‚Äî | Per-socket metadata attached during handshake |

### `/relay` ‚Äî CLI Agent ‚Üî Server

**File:** `relay.ts`

The primary control channel between a running `pi` agent session (TUI) and the relay server. This is the highest-traffic namespace ‚Äî every agent event (heartbeats, message updates, tool calls) flows through here.

**Key client‚Üíserver events:**

| Event | Purpose |
|---|---|
| `register` | Register a new or existing session (sends `cwd`, `sessionName`, optional `parentSessionId`) |
| `event` | Forward an agent event with a sequence number for ordering |
| `session_end` | Signal that a session has ended |
| `session_message` | Send an inter-session message (with optional `deliverAs: "input"`) |
| `session_trigger` | Fire a child‚Üíparent trigger (questions, plans, completion) |
| `trigger_response` | Parent responds to a child trigger |
| `cleanup_child_session` | Request teardown of a completed child session |
| `delink_children` | Sever all child links (e.g. on `/new`) |
| `delink_own_parent` | Child severs its own parent link |
| `exec_result` | Return the result of a remote exec command |
| `get_linked_child_count` | Query the current number of linked child sessions (ack returns `{ ok, count?, error? }`) |

**Key server‚Üíclient events:**

| Event | Purpose |
|---|---|
| `registered` | Confirm registration ‚Äî returns `sessionId`, `token`, `shareUrl`, `isEphemeral`, `collabMode`, `parentSessionId`, `serverTime`, `supportsSessionTriggerAck`, `wasDelinked` |
| `event_ack` | Acknowledge receipt of a sequenced event |
| `connected` | Notify TUI that a viewer connected |
| `input` | Deliver viewer input to the agent (with optional attachments) |
| `model_set` | Instruct agent to switch model |
| `exec` | Remote command execution request |
| `session_trigger` | Deliver a child trigger to the target session |
| `trigger_response` | Deliver a trigger response back to source child |
| `parent_delinked` | Notify child that its parent ran `/new` |
| `session_expired` | Notify that a session has expired |
| `session_message` | Deliver an inter-session message |
| `session_message_error` | Report failure delivering an inter-session message |
| `error` | Generic error |

### `/viewer` ‚Äî Browser ‚Üî Server

**File:** `viewer.ts`

Connects the web UI to a specific session. Viewers receive replayed event history on connect and live events afterwards.

**Key client‚Üíserver events:**

| Event | Purpose |
|---|---|
| `connected` | Viewer greeting that triggers TUI capabilities push |
| `switch_session` | Logically switch the socket to a different session (with generation counter) |
| `resync` | Request a fresh snapshot replay |
| `load_messages` | Request older messages for pagination (paired with `session_messages_page`) |
| `input` | Send user input (collab mode) |
| `model_set` | Switch model (collab mode) |
| `exec` | Remote exec command |
| `service_message` | Forward a service message to the runner |
| `trigger_response` | Human viewer responds to a child trigger (with Socket.IO ack) |
| `mcp_oauth_paste` | Submit a pasted OAuth callback code for MCP servers |

**Key server‚Üíclient events:**

| Event | Purpose |
|---|---|
| `connected` | Confirm viewer attachment ‚Äî includes `lastSeq`, `replayOnly`, `isActive` |
| `event` | Forward an agent event (with optional `replay` flag and `generation`) |
| `disconnected` | Agent disconnected (with reason code) |
| `exec_result` | Forward an exec result back to the viewer |
| `service_message` | Service message from runner |
| `service_announce` | Available services on the connected runner |
| `service_announce_delta` | Incremental delta update for service announcements |
| `error` | Generic error |
| `trigger_error` | Error delivering a trigger response (carries `triggerId` for UI matching) |
| `session_messages_page` | Response to `load_messages` ‚Äî a page of older messages |

<Aside type="tip">
  The `generation` field on viewer events prevents stale events from a previous session leaking into the current view after a `switch_session` call. The viewer discards any event whose generation doesn't match the latest requested switch.
</Aside>

### `/hub` ‚Äî Session List Feed

**File:** `hub.ts`

| Event | Purpose |
|---|---|

| `session_added` / `session_removed` | Session lifecycle |

**Client‚Üíserver events:**

| Event | Purpose |
|---|---|

### `/runners` ‚Äî Runner Feed

**File:** `runners.ts`

A read-only feed that pushes runner daemon status to the web UI. Clients emit no events.

| Event | Purpose |
|---|---|
| `runners` | Full runner list snapshot (sent on connect) |
| `runner_added` | A runner daemon connected |

### Service System

The service system enables runner-side plugins to communicate with the web UI without the relay needing to understand service-specific semantics. Messages are wrapped in a generic `ServiceEnvelope`:

```ts
interface ServiceEnvelope {
  serviceId: string;
  type: string;
  requestId?: string;
  sessionId?: string;  // Attached by relay when forwarding viewer‚Üírunner
  payload: unknown;
}
```

Services can expose UI panels (proxied via WebSocket tunnels) and declare trigger types:

```ts
interface ServicePanelInfo {
  serviceId: string;
  port: number;       // Local port, proxied via tunnel
  label: string;      // Button label in UI
  icon: string;       // Lucide icon name
  panelParams?: Record<string, string>;
}

interface ServiceTriggerDef {
  type: string;        // e.g. "godmother:idea_moved"
  label: string;
  description?: string;
  schema?: Record<string, unknown>;   // JSON Schema for output payload
  params?: ServiceTriggerParamDef[];  // Subscriber-provided config params
}
```

### Tunnel Info

```ts
interface TunnelInfo {
  port: number;
  name?: string;
  url: string;       // Relay tunnel URL fragment
  pinned?: boolean;  // Auto-registered by daemon ‚Äî hidden from UI tunnel list
}
```

### Service Sigils & Trigger Subscriptions

Services can also declare **sigil types** (`ServiceSigilDef`) for inline references rendered in the UI, and **trigger filters** (`TriggerFilter` / `TriggerFilterMode`) so subscribers can restrict which events they receive. Trigger subscriptions are reconciled between server and runner via `TriggerSubscriptionsSnapshot`, `TriggerSubscriptionDelta`, and the `trigger_subscriptions_applied` ack in the `/runner` namespace. Reconciliation is two-way: when a reconnect snapshot is missing subscriptions the runner still has cached (the relay's Redis is ephemeral and is wiped when `pizza web` recreates the container), the runner merges them back locally and re-uploads them via `trigger_subscriptions_restore` so trigger delivery resumes without a runner restart.

## Session Meta-State

**File:** `meta.ts`

The meta-state system tracks ephemeral session UI state ‚Äî todo lists, pending questions, token usage, model info, etc. ‚Äî and synchronizes it from CLI to server to viewers via discrete events.

### `SessionMetaState`

The authoritative shape stored in Redis. Key fields:

| Field | Type | Description |
|---|---|---|
| `todoList` | `MetaTodoItem[]` | Agent's task list |
| `pendingQuestion` | `MetaPendingQuestion \| null` | Active `AskUserQuestion` prompt |
| `pendingPlan` | `MetaPendingPlan \| null` | Active plan awaiting approval |
| `planModeEnabled` | `boolean` | Whether plan mode is on |
| `isCompacting` | `boolean` | Context compaction in progress |
| `retryState` | `MetaRetryState \| null` | API retry error info |
| `pendingPluginTrust` | `MetaPluginTrustPrompt \| null` | Plugin trust prompt |
| `mcpStartupReport` | `MetaMcpReport \| null` | MCP server startup timing/errors |
| `tokenUsage` | `MetaTokenUsage \| null` | Input/output/cache token counts + cost |
| `providerUsage` | `MetaProviderUsage \| null` | Per-provider usage breakdown |
| `thinkingLevel` | `string \| null` | Current thinking level (e.g. "basic", "deep") |
| `authSource` | `string \| null` | Source of the current auth context |
| `model` | `MetaModelInfo \| null` | Current model |
| `goal` | `MetaGoalStatus \| null` | Active session goal (id, description, status, turn/token/cost budgets) |
| `version` | `number` | Monotonic counter for change ordering |

### `MetaRelayEvent`

A discriminated union of 17 event types. The CLI emits these as regular relay events; the server intercepts them (via `isMetaRelayEvent()`), applies them to Redis state (via `metaEventToPatch()`), and broadcasts to hub subscribers.

Event types: `todo_updated`, `question_pending`, `question_cleared`, `plan_pending`, `plan_cleared`, `plan_mode_toggled`, `compact_started`, `compact_ended`, `retry_state_changed`, `plugin_trust_required`, `plugin_trust_resolved`, `mcp_startup_report`, `token_usage_updated`, `thinking_level_changed`, `auth_source_changed`, `model_changed`, `goal_updated`.

## Password Validation

**File:** `password.ts`

Shared password validation used by the server, UI, and CLI. Ensures consistent enforcement of password rules everywhere.

- Minimum 8 characters, maximum 128 characters (scrypt DoS protection)
- At least one uppercase, one lowercase, and one digit
- Uses Unicode codepoint counting (emoji = 1 character)

Exports `validatePassword()` (detailed per-rule results) and `isValidPassword()` (simple boolean).

## Protocol Versioning

**File:** `version.ts`

The `SOCKET_PROTOCOL_VERSION` constant (currently `1`) is sent during Socket.IO handshake. The server checks it via `isSocketProtocolCompatible()` ‚Äî missing or unparseable versions are treated as compatible (graceful degradation), while mismatched integer versions are flagged so the UI can show an update banner.

Also exports semver utilities (`parseSemverTriplet`, `compareSemver`) used for runner version comparisons.

## Extending the Protocol

When adding new events or types:

1. **Add the type in `packages/protocol/src/`** ‚Äî put it in the appropriate namespace file, or `shared.ts` if used across namespaces.

2. **Re-export from `index.ts`** ‚Äî every public type must be re-exported.

3. **Run typecheck across all packages** ‚Äî `bun run typecheck` from the repo root. The server, UI, and CLI all import from `@pizzapi/protocol`, so any breaking change surfaces immediately.

4. **Update both sides** ‚Äî every event needs a sender and a handler. Adding a client‚Üíserver event means updating the server's socket handler. Adding a server‚Üíclient event means updating the consuming component (UI store, CLI handler, etc.).

5. **Bump `SOCKET_PROTOCOL_VERSION`** only for handshake-level breaking changes (new required auth fields, incompatible connection semantics). New events added to existing namespaces do NOT require a version bump ‚Äî unknown events are simply ignored by older clients.

<Aside type="caution">
  The protocol package has **zero runtime dependencies** by design. Do not add imports from `@pizzapi/server`, `@pizzapi/ui`, or any other package. Types flow outward from protocol ‚Äî never inward.
</Aside>
