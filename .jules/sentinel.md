## 2023-10-27 - [IP Spoofing and Brute Force Mitigation on Auth Endpoints]
**Vulnerability:** The registration endpoint blindly trusted the `X-Forwarded-For` header to determine the client IP for rate limiting, allowing attackers to trivially bypass the rate limit by spoofing the header. Additionally, the email sign-in endpoint lacked rate limiting entirely, leaving it vulnerable to credential stuffing and brute force attacks.
**Learning:** In a Node.js `IncomingMessage` to Fetch `Request` adapter pattern, custom headers are a safe way to pass connection-level metadata (like `socket.remoteAddress`) downstream to route handlers, provided that any incoming client headers with the same name are explicitly stripped to prevent spoofing.
**Prevention:** Always extract the direct connection IP (`socket.remoteAddress`) at the edge/server level. Only trust `X-Forwarded-For` if the direct connection originates from a known, trusted proxy (e.g., local loopback). Apply rate limiting to all authentication-related endpoints.
